How does a machine
learn to fix code?

Before LLMs and GitHub Copilot, researchers spent a decade building neural networks that could read and transform source code — one token at a time. The key ingredients: embeddings, sequence models, and attention.

The Big Picture
Source code is discrete, structured text. Neural networks only understand continuous numbers. We need a pipeline: tokenize → embed → model → predict. Each step has deep implications for what the model can and cannot learn.
What We'll Build Up To
A sequence-to-sequence LSTM with attention that takes buggy code as input and outputs the corrected version — the architecture behind early automated program repair systems like DeepFix and SequenceR.

Module 1 · Slide 02

From Tokens to Vectors: Word Embeddings

Neural networks operate on numbers, not text. An embedding is a learned mapping from each token to a dense vector in a high-dimensional space. The geometry of that space encodes meaning.

1

One-hot is too sparse

With a vocabulary of 50,000 tokens, a one-hot vector wastes 49,999 zeros. Every token is equally "distant" from every other — the model can't generalise.

2

Dense embeddings learn geometry

A 128-d embedding vector is learned during training. Tokens that appear in similar contexts end up close together. sort() and order() cluster; while and for cluster.

3

Lookup, not computation

In PyTorch: nn.Embedding(V, d) is just a matrix of shape V × d. Forward pass = row lookup. Back-prop adjusts rows to improve predictions.

Embedding Space (2D projection)
Tokens with similar roles cluster together in embedding space.
Module 1 · Slide 03

Code Embeddings: Code is Different

Source code is not just another kind of text. It has formal syntax, deterministic semantics, and rich structure (ASTs, control flow, data flow). Early work explored how to build embeddings that capture these properties.

🌳
code2vec (Alon et al., 2019)
Represents a method as a bag of AST path-contexts. Each path is a (start leaf → root → end leaf) triple. Paths are embedded and combined with learned attention weights.
🔗
Graph-based Embeddings
Use the data-flow graph to capture which variables influence which. GraphCodeBERT adds data-flow edges to the Transformer attention matrix.
📝
Token-Level Embeddings
Simpler approach: treat code as a sequence of tokens and learn embeddings directly via language modelling. Works surprisingly well at scale (CodeBERT, StarCoder).
Why Code Needs Special Treatment
IDENTIFIER DIVERSITY
Natural language has ~170k English words. A large codebase may have millions of unique identifiers — each method name, variable, class name is potentially unique.
SYNTAX SENSITIVITY
Changing one character can completely change semantics: x = 0 vs x == 0. Natural language is far more robust to small edits.
STRUCTURAL REGULARITY
Code has predictable patterns (loops, conditionals, method signatures) that a model can exploit — but only if the encoding preserves structural information.
Module 1 · Slide 04

The Open-Vocabulary Problem

Word-level models fix a vocabulary at training time. Any token not in the vocabulary becomes <UNK> — a blank. In natural language this is tolerable. In source code, it's catastrophic.

The Core Problem
Source code identifiers follow a power-law / Zipfian distribution. The most common tokens (keywords, operators) appear millions of times. But the vast majority of identifiers appear only once or twice in the entire corpus — and new ones are invented constantly.

Common examples of out-of-vocabulary identifiers:

calculateManhattanDist getUserAuthToken parseXMLResponseV2 AbstractFactoryBean
All become → <UNK>  →  model learns nothing about them
Token Frequency in a Java Corpus
The long tail makes a fixed vocabulary impractical. A 50k vocab still leaves 40-60% of unique tokens as <UNK>.
Module 1 · Slide 05

Early Solutions: Abstraction & Normalization

Before subword tokenization became widespread, researchers tried several heuristics to shrink the effective vocabulary while preserving structure. These techniques helped — but each came with a cost.

Type Abstraction
Replace identifiers with their type placeholders:
String msg = "hi"
→ TYPE VAR = LIT

✅ Tiny vocabulary
❌ Loses all semantic info. Model can't tell what a variable means.
CamelCase Splitting
Split compound identifiers into sub-tokens:
getUserName
→ get User Name

✅ Reuses common sub-words
❌ Still produces OOV parts; order information is lost.
Idiom Mining
Detect frequently recurring code patterns and treat them as single vocabulary entries (Allamanis & Sutton, 2014).

✅ Captures common idioms
❌ Non-exhaustive; domain-specific; doesn't scale to new projects.
The Fundamental Limitation
All of these approaches are lossy pre-processing steps applied before the model sees the data. They reduce vocabulary size by discarding information. The ideal solution should reduce vocabulary without losing information — and generalise to tokens it has never seen. That is exactly what subword tokenization achieves.
Module 1 · Slide 06 · ★ Interactive

Byte-Pair Encoding: Subword Tokenization

BPE starts with a character-level vocabulary and iteratively merges the most frequent symbol pair. The result: a vocabulary of common subword units that can represent any token by composition.

The Algorithm
1. Initialize vocab = all characters
2. Count all symbol pairs in the corpus
3. Merge the most frequent pair → new symbol
4. Repeat until vocab size = V

At inference time, tokenize greedily: match the longest vocabulary entry at each position.
Key Insight for Code
Common programming sub-words (get, set, parse, Error, List, Manager…) appear in BPE vocabulary. Even a completely novel identifier like AbstractNetworkErrorHandler decomposes gracefully.

From RNN to LSTM: The Memory Problem

A basic Recurrent Neural Network (RNN) passes a hidden state from step to step — in theory capturing context from earlier tokens. In practice, gradients vanish exponentially as they flow backward through many timesteps.

Vanilla RNN — Gradient Vanishing
h₁
x₁
→
h₂
x₂
→
h₃
x₃
→
h₄
x₄
→
h₅
x₅
↑ Gradient shrinks exponentially — h₁'s influence on the loss at step 5 ≈ 0
Vanishing Gradient
∂L/∂h₁ = ∂L/∂h₅ × W⁴ (approximately). If the spectral norm of W < 1, this product shrinks to zero. The model forgets early tokens. In code, this means it can't relate a variable declaration to its use 20 tokens later.
LSTM — The Solution
Hochreiter & Schmidhuber (1997) proposed the Long Short-Term Memory cell. Two key changes:

1. Cell state (c) — a separate "memory highway" with near-constant error flow

2. Gates — learnable switches that control what to remember, what to write, and what to output
LSTM Cell at a Glance
σForget gate — what to erase from cell state
σInput gate — what new info to write
tanhCandidate cell — the new content to potentially add
σOutput gate — what to expose as hidden state
Module 1 · Slide 09

Inside the LSTM Cell

At each timestep t, the LSTM takes the current input xt, the previous hidden state ht-1, and the previous cell state ct-1. Four learnable transformations produce a new cell and hidden state.

LSTM Equations
FORGET GATE ft = σ(Wf·[ht-1, xt] + bf)
INPUT GATE + CANDIDATE it = σ(Wi·[ht-1, xt] + bi)
g̃t = tanh(Wg·[ht-1, xt] + bg)
CELL STATE UPDATE ct = ft ⊙ ct-1 + it ⊙ g̃t
OUTPUT GATE + HIDDEN STATE ot = σ(Wo·[ht-1, xt] + bo)
ht = ot ⊙ tanh(ct)
Forget Gate — what to erase
Output ∈ [0,1]. 0 = completely erase the old cell state. 1 = keep everything. If the LSTM has been tracking the subject of a sentence and encounters a period, it may learn to reset.
Input Gate — what to write
Controls how much of the candidate value g̃t to add. The candidate is a new proposed value from tanh; the gate scales it by importance.
Output Gate — what to expose
The cell state may hold many things. The output gate selects what is relevant now to expose as the hidden state ht, which feeds the next timestep and any prediction head.

Module 1 · Slide 11

Training LSTMs: BPTT & Limitations

LSTMs are trained with Backpropagation Through Time (BPTT): unroll the network across all timesteps and apply standard backprop. The cell-state highway mitigates vanishing gradients — but doesn't eliminate all challenges.

BPTT in a Nutshell
1. Forward pass: process the full sequence, recording all gate values and states.
2. Compute loss at each output step (e.g., cross-entropy over next token).
3. Backward pass: propagate gradients back through all unrolled timesteps simultaneously.
4. Accumulate gradients for all weight matrices (Wf, Wi, Wg, Wo).
5. Apply gradient clipping to prevent explosions, then update with Adam/SGD.
Remaining Limitations of LSTMs
Sequential bottleneck: Each timestep depends on the previous hidden state — you can't parallelise across timesteps. Training on long sequences is slow.

Fixed context window via bottleneck: The encoder must compress an entire sequence into a single context vector. For long code files this is lossy.

Still not great at very long dependencies: Even with gating, tracking a variable declared 500 tokens back is difficult.
The Motivation for Attention
What if the decoder didn't have to rely on a single bottleneck vector — but could directly look back at all encoder states with learned weights? That's the key idea behind attention mechanisms.

Attention: The Decoder Looks Back

At each decoding step, the attention mechanism computes a soft alignment over all encoder hidden states. Click an output token below to see where the decoder focuses when producing it.

How It Works
For each decoder step s:
score(s,j) = h_dec[s] · h_enc[j]
α[s,j] = softmax(scores)[j]
context = Σⱼ α[s,j] · h_enc[j]

The context vector is concatenated with the decoder input and fed to the LSTM. This lets the model re-read relevant parts of the input at each step.

Teacher Forcing (Training)
During training, the decoder receives the ground-truth previous token at each step, not its own prediction. This stabilises learning but creates a train/test mismatch — at inference, errors compound.
Beam Search (Inference)
Instead of greedy decoding, maintain the top-k most probable sequences at each step. With beam width 5, the model explores 5 candidate fixes simultaneously and returns the highest-probability complete sequence.

PyTorch: Encoder-Decoder for Bug Fixing

A complete (but concise) PyTorch implementation of the Encoder, Attention module, and Decoder described in this module. This is the architecture at the core of papers like SequenceR and BugLab.

import torch, torch.nn as nn from torch.nn.utils.rnn import pack_padded_sequence, pad_packed_sequence class Encoder(nn.Module): """BiLSTM — reads the buggy token sequence.""" def __init__(self, vocab_sz, emb_d, hid_d, n_layers, drop=0.1): super().__init__() self.embed = nn.Embedding(vocab_sz, emb_d, padding_idx=0) self.lstm = nn.LSTM(emb_d, hid_d, n_layers, batch_first=True, dropout=drop, bidirectional=True) self.proj = nn.Linear(hid_d * 2, hid_d) def forward(self, src, src_len): emb = self.embed(src) # [B,T,E] pack = pack_padded_sequence(emb, src_len, batch_first=True, enforce_sorted=False) out, (h, c) = self.lstm(pack) out, _ = pad_packed_sequence(out, batch_first=True) h_fwd, h_bwd = h[-2], h[-1] h = torch.tanh(self.proj( torch.cat([h_fwd, h_bwd], dim=1))) return out, h # encoder states, summary hidden
class Attention(nn.Module): def __init__(self, hid_d): super().__init__() self.W = nn.Linear(hid_d * 3, hid_d) self.v = nn.Linear(hid_d, 1, bias=False) def forward(self, dec_h, enc_out): T = enc_out.size(1) h = dec_h.unsqueeze(1).repeat(1, T, 1) e = torch.tanh(self.W( torch.cat([h, enc_out], dim=2))) return torch.nn.functional.softmax( self.v(e).squeeze(2), dim=1) # [B,T] class Decoder(nn.Module): def __init__(self, vocab_sz, emb_d, hid_d, n_layers, drop=0.1): super().__init__() self.embed = nn.Embedding(vocab_sz, emb_d, padding_idx=0) self.attn = Attention(hid_d) self.lstm = nn.LSTM(emb_d + hid_d * 2, hid_d, n_layers, batch_first=True, dropout=drop) self.out = nn.Linear(hid_d * 3 + emb_d, vocab_sz) def forward(self, tok, h, c, enc_out): emb = self.embed(tok.unsqueeze(1)) # [B,1,E] aw = self.attn(h[-1], enc_out) # [B,T] ctx = torch.bmm(aw.unsqueeze(1), enc_out) # [B,1,2H] out, (h, c) = self.lstm( torch.cat([emb, ctx], dim=2), (h, c)) pred = self.out(torch.cat([ out.squeeze(1), ctx.squeeze(1), emb.squeeze(1)], dim=1)) return pred, h, c, aw
Instantiate
enc = Encoder(V=32000, E=256, H=512, L=2)
dec = Decoder(V=32000, E=256, H=512, L=2)
optimizer = Adam(list(enc.parameters()) + list(dec.parameters()))
