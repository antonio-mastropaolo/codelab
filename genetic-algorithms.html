Genetic Algorithms · Slide 01

What if we could
evolve the perfect code description?

Nature spent 3.8 billion years optimizing life. Computer scientists borrowed the same recipe — selection, crossover, mutation, repeat — to solve hard optimization problems without ever knowing the answer in advance.

The Core Idea
A Genetic Algorithm maintains a population of candidate solutions. Each generation, the better ones are selected to reproduce and mix their traits — gradually converging toward an optimal answer.

In this module, we'll make this concrete. Instead of abstract bit-strings, our "individuals" are natural language code descriptions. Our fitness measure: code coverage.

Genetic Algorithms · Slide 02

Evolution as Optimization

Biology and computer science use the same vocabulary. Before we write code, let's map the concepts.

🧫
Individual / Chromosome
One candidate solution. In biology: one organism. In our case: one code description.
→ Each description is a natural-language specification that guides the LLM to generate code.
👥
Population
A set of candidate solutions evaluated simultaneously. We run 8 code descriptions per generation.
→ Each generation contains 8 candidate descriptions with different levels of specificity.
💪
Fitness Function
A score measuring how "good" each individual is. Our fitness: code coverage % from a test suite.
→ Survival of the fittest: high coverage = fit to survive.
🔀
Crossover
Two parent solutions combine to create offspring. We split two descriptions at a random word boundary and swap the halves.
→ Like inheriting traits from both parents.
🎲
Mutation
Small random changes to an individual. We append a missing clause (e.g. “Handle duplicates.”) to a code description.
→ Like a copying error in DNA that occasionally creates something better.
🏆
Selection
Which individuals get to reproduce? We use tournament selection: pick 2 at random, keep the fitter one.
→ The strong survive; the weak don't pass on their genes.
Genetic Algorithms · Slide 03

Our Concrete Example:
Evolving Code Descriptions for Code Generation

We want to automatically discover which code descriptions produce the highest-quality code — without manually testing every possibility. That's a search problem. GAs are great at search.

The Setup
Task: Generate a Python function that sorts a list.

Benchmark: A test suite with 12 test cases checks edge cases — empty list, single element, negative numbers, duplicates…

Goal: Find a code description that makes an LLM generate code that passes all 12 tests.
Why GA instead of writing code descriptions manually?
The code description space is combinatorially huge. Even with just 20-word descriptions from a vocabulary of 5,000 words, there are more combinations than atoms in the universe. We need smart search — not brute force.
System Overview
STEP 1 · Input
Population of 8 code descriptions (random initially)
↓
STEP 2 · Evaluate
Send each description to an LLM → collect generated code → measure test coverage
↓
STEP 3 · Evolve
Select, crossover, mutate → new generation
↓ repeat until satisfied ↓

The Fitness Function:
Code Coverage

Fitness is everything. A GA is only as good as its ability to measure "how close" a solution is to the goal. We use test suite coverage as our proxy for code quality.

How Coverage Works as Fitness
1

Generate code

Send the code description to an LLM (or our simulator). Receive a Python function.

2

Run the test suite

Execute all 12 test cases against the generated function using pytest.

3

Measure coverage

Count which lines / branches in the function were exercised by passing tests.

4

Assign fitness score

Coverage % = fitness. 0% = dies. 100% = champion.

Genetic Algorithms · Slide 06

Selection: Survival of the Fittest

Not all code descriptions get to reproduce. Tournament Selection: randomly pick 2 individuals, keep whichever has higher fitness. Repeat until you have a pool of parents.

Tournament Selection — Step by Step
Round 1: #1 (31%) vs #8 (49%) → #8 wins

Round 2: #3 (56%) vs #7 (22%) → #3 wins

Round 3: #6 (71%) vs #2 (44%) → #6 wins

Round 4: #5 (62%) vs #4 (38%) → #5 wins
Why Tournament and not just Top-K?
Taking only the top 2 leads to premature convergence — everyone becomes a clone. Tournament keeps diversity: weaker individuals occasionally make it through, which helps exploration.

Crossover: Mixing Parental Traits

Two parent code descriptions are combined to create two offspring. We split each description at a random word boundary and swap the halves — like gene mixing in biology.

Single-Point Crossover Example
PARENT A
"Write sort_list(arr). Handle: empty list → []," | "single element, negative numbers."
PARENT B
"Python function: takes a list of integers," | "returns sorted copy. Handle empty list."
⬇ Swap after the cut point ⬇
CHILD A
"Write sort_list(arr). Handle: empty list → []," "returns sorted copy. Handle empty list."
Estimated fitness: ~75%
CHILD B
"Python function: takes a list of integers," "single element, negative numbers."
Estimated fitness: ~66%
Key Insight
Child A inherited the "Handle empty list → []" specification clause from Parent A AND the "returns sorted copy" return-type constraint from Parent B — combining useful specification traits from both parents. This is how GAs "build on success."
Genetic Algorithms · Slide 08

Mutation: Exploring New Territory

After crossover, each offspring has a small chance (typically 1–5%) of being mutated. Without mutation, the GA can only recombine what it already has — it can never discover truly new solutions.

Why Mutation Matters
Imagine all 8 code descriptions say "sort integers" — crossover can only produce more "sort integers" variations. Mutation can insert a completely new concept like "handle duplicates" that wasn't in any parent.
Mutation Operators for Code Descriptions
Word substitution: "ascending" → "in ascending order"

Insertion: Add "also handle duplicate values"

Deletion: Remove a less informative clause

Swap: Reorder two sentences
Mutation Example
BEFORE MUTATION (Child A)
"Write sort_list(arr). Handle: empty list → [], single element, returns sorted copy."
Fitness: 75%
↓ mutation: insert clause ↓
AFTER MUTATION
"Write sort_list(arr). Handle: empty list → [], single element, duplicates, returns sorted copy."
Fitness: 83% 🎉 (+8%)
This mutation discovered a new edge case specification that Parent A and B never had — boosting coverage significantly.
Genetic Algorithms · Slide 09

One Complete Generation Cycle

Putting it all together: what happens in a single generation of our Description-GA.

1

Evaluate Population

Send all 8 code descriptions to the LLM, collect code, run test suite, assign coverage fitness. [~3 min in real system]

2

Select Parents

Run 4 tournament rounds. Each selects 1 parent. We get 4 parents total (pairs reproduce). Elitism: always keep the best individual unchanged.

3

Crossover (probability 0.8)

Pair parents, apply single-point crossover at a random position. 2 pairs → 4 children. 80% chance of crossing; otherwise children = copies of parents.

4

Mutation (probability 0.05 per word)

For each word in each child: 5% chance to substitute, insert, or delete. Keeps diversity alive and explores new description structures.

5

Form New Generation

4 children + 1 elite + 3 selected survivors = new population of 8. Repeat from step 1.

Reading the Fitness Chart

The fitness chart tells the story of the search. Knowing how to interpret it is key to understanding when a GA is working — and when it's stuck.

What to Look For
Rapid early gains: Best fitness often jumps fast in the first few generations as obvious improvements are found quickly.

Plateau: After ~gen 8–10, gains slow down. The "easy wins" are taken; harder optimizations require more generations or specific mutations.

Best vs Average gap: A wide gap means high diversity — good for exploration. A narrow gap means the population is converging.
Warning Signs
Premature convergence: Average and best lines merge early, then plateau below 100%. GA got "stuck" — all individuals too similar.

Fix: Increase mutation rate, increase population size, or use multi-restart.

GA vs Other Optimization Methods

GAs aren't magic. When should you use one versus a different approach?

🔀
Genetic Algorithm (our approach)
Good when: search space is huge, fitness landscape is rugged (many local optima), parallel evaluation is cheap.
✅ Description search: huge space, rugged landscape, easy to parallelize.
🏔️
Hill Climbing / Gradient Descent
Good when: the fitness function is smooth and differentiable. Gets stuck in local optima easily.
⚠️ Description search: not differentiable — can't take gradient of text.
🔭
Exhaustive / Brute Force
Good when: search space is small. Guarantees finding the optimum — but only feasible at tiny scale.
❌ Description search: space is astronomically large. Completely infeasible.
🎲
Random Search
Good when: you have no structure to exploit and need a quick baseline. Surprisingly competitive in high dimensions.
⚠️ Description search: okay baseline but GA consistently beats it given enough generations.
🌡️
Simulated Annealing
Good when: single-individual search with temperature-controlled acceptance of worse solutions to escape local optima.
✅ Description search: works well but slower than GA — no parallel population.
🤖
Reinforcement Learning
Good when: you can define a reward signal and need sequential decision-making. Needs lots of samples.
✅ Description search: RLHF (Reinforcement Learning from Human Feedback) is exactly this — but needs more data.

Key Takeaways

01 · Population Thinking

Maintain Diversity

GAs work because they search many regions simultaneously. A single best solution is not enough — diversity in the population enables exploration.

02 · Fitness is Everything

Define Success Precisely

The GA is only as good as its fitness function. Vague fitness → vague convergence. Code coverage is a precise, measurable proxy for description quality.

03 · Exploration vs. Exploitation

Balance is Key

Selection and crossover exploit known good solutions. Mutation explores unknown territory. Too much of either breaks the algorithm.

04 · Emergent Solutions

Discovering the Unexpected

GA-evolved code descriptions often look different from human-written ones — they discover unintuitive structures that happen to maximize coverage.

05 · GA in Software Engineering

Beyond Description Evolution

GAs are used for test case generation, code optimization, hyperparameter tuning, scheduling problems, and architectural search.

06 · The Vibe Coding Connection

Systematic Description Engineering

Vibe coding + GAs = systematic description engineering. Instead of guessing code descriptions, we evolve them automatically — turning intuition into optimization.
