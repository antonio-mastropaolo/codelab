Self-Attention: The Core Idea

In self-attention, each token asks: "which other tokens in this sequence are most relevant to understanding me?" The answer is computed via three learned projections: Query, Key, and Value.

Query — "What am I looking for?"
Each token projects itself into a query vector Q = xWQ. This represents what information this token needs from others.
Key — "What do I offer?"
Each token also projects to a key vector K = xWK. Keys describe what information this token contains, available to be retrieved by other tokens' queries.
Value — "What do I contribute?"
V = xWV holds the actual content to pass forward. The output at each position is a weighted sum of all values, where weights come from Q·Kᵀ similarity.
The Scaled Dot-Product Attention
Q = X · WQ    K = X · WK    V = X · WV
↓
scores = Q · Kᵀ / √dk
↓ softmax →
weights = softmax(scores)
↓
output = weights · V
√dk scaling prevents dot products from growing too large, which would push softmax into low-gradient regions.

Multi-Head Attention

A single attention head can only capture one type of relationship. Multi-head attention runs h attention heads in parallel — each with independent WQ, WK, WV projections — then concatenates and projects the results.

Multi-Head Mechanism
Head 1
syntax patterns
Head 2
arg–param links
Head 3
coreference
Head 4
data flow
↓ concat
Concat([Head1, Head2, Head3, Head4]) · WO
Parameter Efficiency
With h heads and dimension d, each head uses d/h dimensions. Total compute is similar to a single head at dimension d — but the model gets h independent perspectives on the same input.
What Different Heads Learn (Empirically)
Positional heads: attend to tokens immediately before/after (local syntax)
Bracket heads: ( attends to ) and vice versa across any distance
Coreference heads: a variable use attends back to its declaration
Rare token heads: unknown subwords attend to semantically related tokens to borrow meaning
Typical Scale
BERT-base: 12 heads × 64d = 768d. CodeBERT: same. GPT-4: reportedly 128 heads at very high dimension across 96+ layers. More heads → more specialisation.

The Transformer Architecture

The Transformer stacks N identical layers. Each layer is a Multi-Head Attention block + Feed-Forward Network, both wrapped with residual connections and layer normalisation.

One Encoder Layer
Multi-Head Self-Attention
+ residual  →  LayerNorm
Feed-Forward Network (2 layers, ReLU)
+ residual  →  LayerNorm
Output representations (same shape as input)
Stacked N=6–96 times depending on model size.
Residual Connections
output = LayerNorm(x + sublayer(x))

The residual connection adds the original input to the sublayer output. This creates a highway for gradients: even if a sublayer learns nothing, the gradient flows directly through the addition. This is what enables training 100+ layer Transformers.
Feed-Forward Network (FFN)
Each position independently passes through: FFN(x) = max(0, xW₁+b₁)W₂+b₂. Hidden dimension is typically 4× the model dimension. The FFN acts as a key-value memory that stores factual associations learned from training data.
Encoder vs Decoder
The encoder uses bidirectional self-attention (all tokens see all). The decoder uses masked self-attention (each token only attends to previous tokens) plus cross-attention to encoder outputs.

Positional Encoding: Injecting Order

Self-attention is permutation-invariant: if you shuffle the input tokens, the attention scores don't change (only the output positions swap). Transformers must be given explicit position information.

Sinusoidal Positional Encoding (original)
PE(pos, 2i) = sin(pos / 100002i/d)
PE(pos, 2i+1) = cos(pos / 100002i/d)

Each position gets a unique pattern of sin/cos at different frequencies. Added directly to the token embedding before the first layer.
Learned Positional Embeddings
BERT and most modern code models use learned positional embeddings — one trainable vector per position. Slightly better in practice but can't generalise beyond the maximum training length.
RoPE & ALiBi (Modern)
Recent models (Llama, CodeLlama) use Rotary Position Embeddings (RoPE) which encode relative position directly in the Q·K dot product, enabling better length extrapolation.

Each row = a position. Each column = one embedding dimension. Colors show sin/cos wave values from −1 (dark) to +1 (bright green).

LSTM vs Transformer: Head to Head

Both architectures can learn representations of sequences — but their inductive biases, scaling properties, and practical trade-offs differ significantly.

DimensionLSTMTransformer
Processing Sequential — must process t before t+1 Parallel — all positions processed simultaneously
Long-range dependencies Indirect — must flow through all intermediate cells Direct — any token attends to any other in O(1) steps
Compute complexity O(T) per layer; easily handles very long sequences O(T²) per layer (attention matrix); quadratic in sequence length
Memory (inference) O(1) per step (just the hidden state) O(T) — must cache all prior key-value pairs
Pre-training at scale Limited — hard to scale; sequential bottleneck Excellent — trivially parallelises across GPUs/TPUs
Small datasets Competitive — fewer parameters, strong inductive bias Data-hungry without pre-training; overfits easily
Streaming/online Natural — processes one token at a time Requires KV cache or re-computation at each step
The Practical Verdict
For research-scale SE automation (bug fixing, code summarisation, clone detection) today: Transformers dominate, primarily because large pre-trained code models are freely available. LSTMs remain relevant in resource-constrained settings, streaming inference, and scenarios where training data is small.

Pre-Training: Learning from the Unlabeled

The key insight behind BERT, GPT, and their code variants: you can learn powerful representations from raw, unlabeled text using self-supervised objectives — without any human annotation.

🎭
Masked Language Modelling (MLM) — BERT-style
Randomly mask 15% of tokens. Train the model to predict the masked tokens from context. Forces the model to learn bidirectional language understanding. Used by: BERT, CodeBERT, RoBERTa.
🔮
Causal Language Modelling (CLM) — GPT-style
Train to predict the next token given all previous tokens (left-to-right). Naturally suited for generation. Used by: GPT-2/3/4, Codex, StarCoder, CodeLlama.
🔁
Span Denoising — T5-style
Corrupt the input by masking contiguous spans. Train to reconstruct the original. The encoder-decoder architecture naturally learns to repair text. Used by: CodeT5, T5.
Pre-Training Pipeline
DATA
GitHub: 54M+ repos — Python, Java, JS, Go, Rust… Raw code, no labels needed.
↓
SELF-SUPERVISED OBJECTIVE
Predict masked tokens / next tokens — no human annotation required.
↓ 100B-1T tokens ↓
RESULT
Model learns: syntax, idioms, common patterns, API usage, variable naming conventions — without being told what any of it means.

Fine-Tuning: Adapting to Downstream Tasks

After pre-training, the model has rich representations but no task-specific head. Fine-tuning adds a small task head and continues training on a (typically small) labeled dataset.

Two Strategies
Full fine-tuning: update all parameters. Best accuracy but requires storing one model copy per task. Used when compute is plentiful.

Frozen backbone + head: freeze the pre-trained encoder, only train the classification head. Much cheaper. Works well when labeled data is small (100–1000 examples).
Why Labeled Data is Scarce in SE
Creating labeled SE datasets requires developer expertise:
— Classifying a bug report to a component needs project knowledge
— Identifying a code smell requires understanding code quality
— Annotating a patch as correct requires running tests

A large project might have 10k labeled bug reports after years of effort — but GitHub has billions of unlabeled lines of code. Pre-training exploits the unlabeled majority.

Code Models: The Ecosystem

A new generation of Transformer-based models pre-trained specifically on source code has emerged, each with a different architecture, training objective, and intended use case.

ModelOrgArchitecturePre-train ObjectiveBest For
CodeBERT Microsoft Encoder-only (BERT) MLM + RTD (bimodal: code + docstring) Classification, retrieval, embeddings
GraphCodeBERT Microsoft Encoder-only + data-flow MLM + edge prediction on DFG Code search, clone detection
CodeT5 Salesforce Encoder-Decoder (T5) Span denoising + identifier tagging Code generation, summarisation
StarCoder2 Hugging Face Decoder-only (GPT) CLM on 600+ languages (The Stack v2) Code completion, generation
CodeLlama Meta Decoder-only (Llama 2) CLM + infilling (code only) Completion, infilling, instruction following
Which to Choose?
For classification and embedding extraction (bug triaging, clone detection, vulnerability detection): encoder-only models like CodeBERT / GraphCodeBERT are the standard choice. For generation tasks (code completion, repair): decoder-only models (StarCoder, CodeLlama) produce better output. Encoder-decoder models (CodeT5) occupy a middle ground — strong at both.

Code Embeddings from Transformers

After fine-tuning (or even without it), the [CLS] token's output from an encoder is a rich embedding of the full input. These embeddings cluster semantically similar inputs together. Click a component label below to highlight its bug-report cluster.

Contextual vs Static Embeddings
In word2vec, null always has the same vector. In BERT/CodeBERT, null in a Java null-check has a different embedding than in a JSON response — the context shapes the representation. This massively improves downstream task performance.
[CLS] Pooling
BERT prepends a special [CLS] token. After the final layer, its hidden state attends to the entire sequence and produces a fixed-size (e.g. 768-d) vector representing the whole input. Add a linear layer on top → classifier.

Bug Triaging: From Embeddings to Labels

Large open-source projects receive hundreds of bug reports per week. Bug triaging — assigning each report to the right developer or component — is labour-intensive. Embeddings + classification make it automatable.

The Scale Problem
The Eclipse IDE bug tracker has received over 300,000 bug reports across 15+ components (UI, JDT, Platform, CDT…). Mozilla Firefox has similar numbers. Manually routing all of these to the right team is a full-time job per project.
Input Features
Bug report title: "NPE in UIManager.paintComponent when dark theme"
Description: stack traces, steps to reproduce, version info
Code context (optional): snippet triggering the bug

HuggingFace: Pre-Train → Embed → Classify

A complete, runnable example: load CodeBERT, freeze the backbone, attach a classification head, and fine-tune on bug-triaging data.

from transformers import AutoTokenizer, AutoModel import torch, torch.nn as nn # ── 1. Load pre-trained CodeBERT ────────────── tokenizer = AutoTokenizer.from_pretrained( "microsoft/codebert-base") backbone = AutoModel.from_pretrained( "microsoft/codebert-base") # Freeze backbone weights for p in backbone.parameters(): p.requires_grad = False # ── 2. Classification head ───────────────────── class BugTriageModel(nn.Module): def __init__(self, encoder, n_classes, drop=0.1): super().__init__() self.enc = encoder self.drop = nn.Dropout(drop) self.head = nn.Linear(768, n_classes) def forward(self, ids, mask): out = self.enc(input_ids=ids, attention_mask=mask) cls = out.last_hidden_state[:, 0, :] # [CLS] return self.head(self.drop(cls)) COMPONENTS = ["Platform/UI", "JDT/Core", "Platform/Debug", "CDT/Core", "Platform/Runtime", ...] model = BugTriageModel(backbone, n_classes=len(COMPONENTS))
# ── 3. Training loop ─────────────────────────── opt = torch.optim.Adam( filter(lambda p: p.requires_grad, model.parameters()), lr=2e-4) loss_fn = nn.CrossEntropyLoss() for epoch in range(10): for title, label in dataloader: enc = tokenizer(title, return_tensors="pt", max_length=128, truncation=True, padding=True) logits = model(**enc) loss = loss_fn(logits, label) loss.backward() opt.step(); opt.zero_grad() # ── 4. Inference ─────────────────────────────── model.eval() report = "NPE in UIManager.paintComponent — dark theme" enc = tokenizer(report, return_tensors="pt", max_length=128, truncation=True) with torch.no_grad(): logits = model(**enc) pred = logits.argmax(dim=-1) print(f"→ {COMPONENTS[pred]}") # → Platform/UI
Key Takeaways
1. Pre-trained code models are plug-and-play via HuggingFace.
2. Frozen backbone + small head trains in minutes on modest hardware.
3. The same recipe works for vulnerability detection, code smell classification, duplicate bug detection — just change the labels.
4. For generation tasks, swap AutoModel for AutoModelForCausalLM.
